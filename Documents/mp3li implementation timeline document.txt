Live Deezer Enrichment Pivot + CD Art Placement Rule + Visible-Only Loading Strategy (May 9, 2026)
- Objective:
  - Stop treating additional song metadata as an all-or-nothing “finish a full preload pipeline first” requirement for the current project milestone.
  - Keep my `v2` additional-data getter work meaningful for later iterations, but change the current app implementation to a route I could actually finish, stabilize, and test within the remaining time.
  - Make the app show real song metadata and real art now, while still preserving a future path where fuller pre-organized data can be generated by my tools and used later if I want that architecture.

Why I made this switch:
- I do still plan to finish doing full runs with my `v2` data getter in the future.
- I still want future iterations of the app to have the option of using more fully pulled/organized song data rather than depending only on live enrichment.
- However, for this phase of the project I deliberately changed direction because:
  - time left in the project was becoming a real constraint,
  - I am already comfortable working with external APIs directly,
  - Deezer already exposes most of the extra display metadata I immediately needed,
  - the app did not actually need every possible extra field pre-written into the database in order to improve the user-facing experience right now,
  - and preview URLs were always going to be awkward to treat as permanent DB fields anyway because they expire.
- So the practical decision became:
  - for now, use the live external Deezer API instead of requiring a full “pull everything with my `v2` tool and write it into the database first” workflow,
  - but keep the `v2` tool untouched because I still expect it to matter for future iterations.

What this pivot means and what it does not mean:
- It means:
  - the app now resolves most additional song display metadata live when songs are actually needed on screen.
- It does not mean:
  - I abandoned my `v2` tool,
  - I no longer care about fuller organized data,
  - or I concluded that live-only enrichment is the final forever architecture for the project.
- This was a milestone-driven implementation pivot, not a “delete the old direction and pretend it never mattered” change.

What I explicitly did not change:
- I did not rewrite or delete my tools.
- I kept `tools/` untouched on purpose.
- I did not convert the app to use lyrics pulling in this route.
- I did not implement language extraction in this route.
- I did not bulk-insert all of the new Deezer metadata into the shared project database.
- I did not treat preview URLs as permanent truth because they expire.
- I did not move Deezer API calling into the frontend.

Core architecture after the pivot:
- The project database is still the source of the base song lists.
- The backend still calls the existing stored procedures to get those base song rows.
- After the backend gets those rows, it enriches each song on demand with Deezer data.
- The frontend still talks to the app backend only.
- The frontend does not talk to Deezer directly.
- The backend caches successfully resolved Deezer metadata locally so repeated requests can be answered much faster after the first resolution.
- The frontend receives song objects that now contain:
  - the original DB-driven song values,
  - plus the extra Deezer metadata the backend was able to resolve for that song.

Why the backend owns Deezer instead of the frontend:
- Keeps external API logic out of browser/mobile code.
- Keeps app-side secret handling centralized.
- Gives me one place to rate-limit, retry, cache, and fall back safely.
- Lets web and mobile both benefit from the same enrichment logic through normal app endpoints.
- Prevents the frontend from having to orchestrate multiple raw Deezer requests for every song card/list item.

How the backend resolves a song now:
1. A screen asks the backend for a country/profile/hidden-gem song payload.
2. The backend gets base song rows from the existing stored procedure.
3. For each song row, the backend tries to resolve Deezer metadata using song name + artist name.
4. Before hitting Deezer, it checks the local cache first.
5. If the song is already cached and still usable, the backend reuses that cached metadata.
6. If it is not cached or needs refresh, the backend searches Deezer and then enriches the song with the resolved data.
7. The resolved metadata is written back into the local cache so later requests can reuse it.
8. The enriched song object is returned to the frontend.

Important unresolved-song rule:
- If the backend cannot resolve a usable Deezer result for a song, that song is not returned as a “broken” item.
- Instead:
  - it is skipped,
  - and the backend continues farther into the same stored-procedure result stream to try to fill the visible UI count with usable songs.
- This was done intentionally so the user would not see broken song rows with empty Deezer-dependent metadata.

Unresolved retry protection:
- Unresolved songs are not hammered repeatedly.
- The service has:
  - `UnresolvedRetryWindow = 12 hours`
- Meaning:
  - if a song was just marked unresolved, the backend will not immediately keep re-trying it on every request within that window.

Cache design:
- Local cache folder:
  - `backend/Capstone.API/live_song_enrichment_cache/`
- Files written there:
  - `live_song_cache.json`
  - `live_song_cache.csv`
  - `live_song_cache.txt`
- That folder is intentionally gitignored:
  - `.gitignore` includes `backend/Capstone.API/live_song_enrichment_cache/`

What each cache file is for:
- `live_song_cache.json`
  - the main working cache snapshot
  - this is the file the backend loads into memory
- `live_song_cache.csv`
  - readable export of resolved entries
  - useful for inspection / scanning / spreadsheet-like review
- `live_song_cache.txt`
  - readable block-style export of resolved entries
  - useful when I just want to inspect the data quickly in plain text

How the cache works in practice:
- On first service use, the backend loads `live_song_cache.json` into memory as `_cacheByKey`.
- After that, normal cache hits are in-memory lookups first.
- That means the backend is not re-reading the file from disk for every single song request.
- After new data is resolved, the service persists the updated snapshot back out to JSON/CSV/TXT.

Why this matters for speed:
- A cache hit is much faster than a live Deezer call.
- The true slow path is:
  - first-time enrichment,
  - preview refresh,
  - or a Deezer retry/fallback path during transient failures.
- So the design goal became:
  - cached songs should feel fast,
  - uncached songs can temporarily show loading UI,
  - and the app should never feel empty just because live enrichment is still happening.

Preview URL handling:
- Preview URLs are deliberately treated differently from the rest of the metadata.
- Reason:
  - the preview URL itself expires
  - so it cannot be treated as permanent truth the same way the rest of the metadata can
- Current preview behavior:
  - store the preview URL when resolved
  - parse its expiry time
  - reuse it while it is still valid
  - refresh it by known Deezer track id when it expires
  - avoid serving an obviously expired preview as if it were still fresh

Exact preview expiry parsing rule:
- The backend reads the `hdnea` query parameter from the preview URL.
- It looks inside that value for the `exp=` segment.
- That Unix timestamp is converted into `PreviewExpiresAtUtc`.

What “keep preview URLs temporarily” means in this implementation:
- If the user plays the same song again and the preview URL is still valid:
  - reuse it from cache
  - do not re-ping Deezer unnecessarily
- If the preview is expired:
  - refresh it through the backend
- If refresh fails and the old preview is expired:
  - do not keep pretending the expired preview is valid

Rate-limit rule I implemented:
- Treat Deezer as:
  - `50 requests every 4.9 seconds`
- Code constants:
  - `DeezerMaxRequests = 50`
  - `DeezerWindow = 4.9 seconds`
- Enforcement mechanism:
  - sliding-window rate limiter in the enrichment service

Reason I used `4.9` instead of `5.0`:
- The goal was to stay slightly inside the limit rather than riding the exact edge of it.

Failure/retry hardening added after real backend errors:
- Real issue encountered during testing:
  - Deezer transport/TLS requests were sometimes failing with:
    - `HttpRequestException`
    - `IOException`
    - `SocketException (54)`
    - `Connection reset by peer`
- Problem before hardening:
  - those transient external failures could bubble up and break country-song endpoint responses
- Hardening added:
  - retry transient Deezer failures up to `3` attempts
  - retry delays:
    - `250ms`
    - `800ms`
    - `1600ms`
  - treat `404` as a normal “not found / no data” outcome
  - if refresh/search fails transiently and there is already cached resolved data:
    - return safe cached metadata instead of failing the whole request
  - if fallback cache still contains an expired preview:
    - strip that expired preview from the safe fallback entry instead of serving a dead preview URL

How Country Profile now uses live enrichment:

  - `CountryRepository.GetCountryProfileAsync`
- Stored procedure:
  - `sp_GetCountryProfile`
- Result-set behavior:
  - set `0` = summary stats

How Country Hidden Gems Preview now uses live enrichment:
- Backend route:
  - `GET /api/country/{countryCode}/hidden-gems/preview?year={year}&limit={limit}`
- Repository:
  - `CountryRepository.GetHiddenGemsPreviewAsync`
- Stored procedure:
  - `sp_GetCountryHiddenGemsPreview`
- Important backend behavior:
  - requested preview count is over-fetched from SQL first so unresolved songs can be skipped
  - raw limit formula:
    - `min(max(requestedLimit * 4, requestedLimit + 20), 100)`
- Then for each raw row:
  - enrich through Deezer
  - skip unresolved songs
  - dedupe by normalized `song + artist`
  - stop once the requested number of usable unique preview items has been collected

Why that hidden-gems preview over-fetching was necessary:
- If some songs fail Deezer resolution and I only fetched the exact visible count from SQL, the preview strip could come back too short or broken.
- Over-fetching lets the backend find enough usable preview songs while still keeping the UI count clean.

Dedupe protection added for hidden-gems preview:
- Requirement:
  - hidden-gems preview strips on Country page and Comparison View should never show duplicates
- Backend protection:
  - `CountryRepository.GetHiddenGemsPreviewAsync` dedupes by normalized `song + artist`
- Frontend protection:
  - preview mapping/rendering logic also dedupes again as a second safeguard

How Country Songs paging now uses live enrichment:
- Backend route:
  - `GET /api/country/{countryCode}/songs?year={year}&listType={shared|unique}&page={page}&pageSize={pageSize}`
- Repository:
  - `CountryRepository.GetCountrySongsPageAsync`
- Stored procedure:
  - `sp_GetCountrySongsPaged`
- Backend page-size clamp:
  - `1..100`
- Important behavior:
  - the backend does not just trust the raw DB count as the final usable song count anymore
  - it scans raw batches, enriches songs, skips unresolved ones, and computes the page from the resolved set
- Raw scan batch size used:
  - `100`
- Result:
  - `Items` are usable enriched songs only
  - `TotalCount` is the resolved usable count, not the raw row count
  - `HasMore` is based on the resolved usable count

Why the country-song paging had to be handled that way:
- Once unresolved Deezer songs are skipped, raw SQL row counts and visible usable song counts are no longer the same thing.
- If I had left paging based only on raw DB rows, the frontend could ask for a page and get too few visible usable songs or inconsistent “has more” behavior.

Important UI/loading philosophy I locked in:
- I never wanted big sections of the app to look blank just because Deezer enrichment was still in progress.
- The preferred behavior became:
  - render the structure immediately
  - show whatever DB/profile content is already available
  - use `Loading...` text where Deezer-dependent text is not ready yet
  - use the existing placeholder colors where art is not ready yet
  - put a centered spinner on the art slot while the art is loading
- In other words:
  - visible placeholders and progressive loading were preferred over empty holes and stalled-looking sections

Problem with the first live-enrichment version:
- It was still too eager about some song-page requests.
- Country/Comparison screens could wait too long on larger follow-up song loads.
- Major sections could remain in `Loading...` longer than acceptable even though enough visible data already existed to render a meaningful first state.

Visible-only loading / scroll-driven strategy I switched to:
- I explicitly did not want the app to request metadata for songs the user could not even see yet.
- So the desired rule became:
  - initial render should use the first visible content that is already available
  - deeper long-list enrichment should happen only when the user scrolls enough to need more

  - Use the following SQL definition:

```sql
USE HiddenGemMusic;
GO

CREATE OR ALTER PROCEDURE sp_GetDiscoverPageInfo
    @Year INT
AS
BEGIN
    SET NOCOUNT ON;

    SELECT
        c.iso_code AS country_code,
        c.full_name AS country_name,
        c.latitude,
        c.longitude,
        c.region,
        ISNULL(hg_agg.hidden_gem_count, 0) AS hidden_gem_count,
        top_song.top_album_name AS top_album_name,
        top_song.top_artist_name AS top_artist_name
    FROM Country c
    LEFT JOIN (
        SELECT
            country_id,
            COUNT(*) AS hidden_gem_count
        FROM HiddenGems
        WHERE chart_year = @Year
        GROUP BY country_id
    ) hg_agg ON hg_agg.country_id = c.country_id
    LEFT JOIN (
        SELECT
            ce.country_id,
            COALESCE(NULLIF(s_inner.album_name, ''), s_inner.title, 'Unknown Song') AS top_album_name,
            COALESCE(NULLIF(a_inner.artist_name, ''), 'Unknown Artist') AS top_artist_name,
            ROW_NUMBER() OVER (
                PARTITION BY ce.country_id
                ORDER BY COUNT(*) DESC, ce.song_id ASC
            ) AS rn
        FROM ChartEntry ce
        JOIN DIM_Song s_inner ON s_inner.song_id = ce.song_id
        LEFT JOIN Bridge_SongArtist bsa_inner
            ON bsa_inner.song_id = s_inner.song_id
           AND bsa_inner.artist_order = 1
        LEFT JOIN DIM_Artist a_inner ON a_inner.artist_id = bsa_inner.artist_id
        WHERE ce.country_id IS NOT NULL
          AND YEAR(ce.snapshot_date) = @Year
        GROUP BY ce.country_id, ce.song_id, s_inner.album_name, s_inner.title, a_inner.artist_name
    ) top_song ON top_song.country_id = c.country_id AND top_song.rn = 1
    WHERE c.latitude IS NOT NULL
      AND c.longitude IS NOT NULL
      AND c.iso_code IS NOT NULL
      AND LEN(c.iso_code) = 2
      AND c.iso_code NOT LIKE '%[^A-Z]%'
    ORDER BY c.full_name;
END;
GO
```
