Genius Web Matching Upgrade Pass + Confirmed Real-World URL Patterns + v1/v2 Tool Naming Cleanup (May 9, 2026)
- Objective:

  - Make `Smart Catch-Up` meaningfully revisit older skipped rows from the beginning when new URL-attempt logic is added.

Why this follow-up pass happened:
- After getting `v2` running much faster and more stably, started spot-checking skipped songs manually in Google/Genius to see whether the remaining misses were just “Genius truly does not have this song” or whether my URL-construction logic still needed improvement.
- That manual checking immediately exposed additional concrete Genius URL patterns that the current fallback logic still was not handling well enough.
- So instead of leaving those as vague suspicions, used the manually verified examples to improve the matching rules directly.

Important manual URL confirmations from real skipped songs:
- Confirmed `feat.` stripping pattern:
  - Real Genius URL found manually:
    - `https://genius.com/Bob-airplanes-lyrics`
  - The skipped row that motivated this confirmation was a case where the song’s displayed title still included featured-artist metadata.
  - This confirmed that for Genius URL construction, featured metadata should often be removed completely from the title rather than preserved in the slug.
- Confirmed clean-title pattern:
  - Real Genius URL found manually:

  - Skipped row:
    - `Alaska (feat. Priestess)` by `Gemitaiz`

    - `Alarm` by `Lookas, Krewella`
  - This confirmed that some collaboration rows need a full-artist slug where the artists are joined with `and`, not just comma-flattened into a simple hyphen chain and not always reduced to the first artist alone.

- The previous fallback logic had improved a lot already, but it still had two important problems:
  1. It could stop too early on a page-text mismatch.
  2. It still did not explicitly try a full multi-artist slug joined with `and` for rows that list multiple artists.

Specific matching logic changes made in this pass:
- Page-text mismatch no longer stops the fallback chain immediately:
  - Previously, a result like:
    - `Constructed Genius URL page text did not match song+artist`
  - could short-circuit later fallback attempts.
  - That was a problem because a wrong first guess should not automatically prevent trying a better clean-title or artist-variant guess.
  - Changed the logic so deterministic mismatch now behaves more like a recoverable candidate failure rather than an immediate end-state for the whole row.
- More aggressive `feat.` cleanup now matters even more:
  - The clean-title path already removed featured metadata, but the manually confirmed examples strengthened the case that this is not just a “nice maybe” fallback.
  - It is a real recurring Genius URL pattern.
- Full multi-artist slug fallback was added:
  - Rows like:
    - `Lookas, Krewella`
  - can now produce a fallback candidate more like:
    - `lookas-and-krewella`
  - instead of only:
    - flattened comma forms
    - or first-artist-only forms
- Existing punctuation strategy expansion remained in place:
  - the tool still keeps both:
    - old punctuation-stripping behavior
    - and newer punctuation-as-separator behavior
  - so titles like `Heal'lang` can still try both:
    - `heallang`
    - and `heal-lang`

Why these changes matter to Smart Catch-Up:
- Once added new candidate labels and new slug patterns, the old skipped rows could no longer be treated as “fully exhausted” in the same way as before.
- A row that had previously consumed the older attempt set might still have brand-new valid attempts available under the expanded logic.
- Because `Smart Catch-Up` classifies rows by:
  - existing `lyrics_url`
  - prior `attempt_trail`
  - and currently available candidate labels
- adding new attempt labels means older skipped rows now naturally show up as still having remaining useful work left.
- In practical terms:
  - this means `Smart Catch-Up` should be restarted and allowed to scan from the beginning again,
  - not because the whole system forgot progress,
  - but because the matching vocabulary itself became richer and older skips need to be reevaluated against the new attempt set.

Smart Catch-Up mode added for Genius Web only:
- Then realized the old `retry skipped` behavior was too blunt.
- Once the tool had already failed a full deterministic attempt set, it was wasteful to keep retrying those same rows forever.
- Also, once the tool already had a real `lyrics_url` for a song, it should not waste time reprocessing that row.
- Added a new Genius Web-only mode:
  - `Smart Catch-Up`
- What this mode does:
  - skips rows that already have a stored Genius `lyrics_url`
  - revisits skipped rows only if they still have untried fallback attempts left
  - avoids rerunning the already-failed first slug if that exact deterministic attempt has already been consumed
  - automatically continues into normal forward progress for never-before-tried songs once the “useful skipped backlog” has been handled
  - treats rows as fully exhausted once all deterministic Genius Web attempts have already failed cleanly
  - still leaves room to retry transient/infrastructure-style failures later
- This was important because it made the background Genius Web run much more intelligent:
  - it stopped wasting time on already-solved rows,
  - stopped repeatedly hammering fully exhausted deterministic misses,
  - and still kept moving the run forward.

Aggressive throughput tuning work:
- After the logic-side improvements, pushed the throughput more aggressively because the whole point was to leave this background process running while other work continued.
- Increased the worker count in stages:
  - first `3`
  - then `5`
  - then `10`
- Matched the shared request cap upward while those limiters were still active.
- Then, because the real bottleneck still did not appear to be pure request throttling anymore, disabled the explicit Genius Web limiter entirely for that path.
- At that point the tool still was not hitting the throughput the goal was.
- That reinforced the conclusion that the repeated file rewrites were the real slowdown.

Batch-write redesign for the Genius Web path:
- To address that, changed the Genius Web path so it no longer writes full runtime/output files after every completed song.
- Instead, the Genius Web path now batches writes every `10` processed rows.
- Specifically, it now buffers and flushes in chunks of `10` for:
  - `run_state.json`
  - `rows_cache.json`
  - `enriched_songs.csv`
  - `enriched_songs.json`
  - `enriched_songs.txt`
- Leftover buffered work still flushes at chunk end.
- Forced-stop / backoff paths still do an immediate forced flush.
- Tradeoff accepted on purpose:
  - if the process is interrupted between flushes, up to `9` newest Genius Web completions can be lost from disk
  - but the repeated per-song write stall is much lower
- This was a deliberate throughput tradeoff because the whole point of this pass was to make the background run actually practical over long unattended periods.

Live Deezer Enrichment Pivot + CD Art Placement Rule + Visible-Only Loading Strategy (May 9, 2026)
- Objective:
  - Stop treating additional song metadata as an all-or-nothing “finish a full preload pipeline first” requirement for the current project milestone.
  - Keep the `v2` additional-data getter work meaningful for later iterations, but change the current app implementation to a route that could actually be finished, stabilized, and tested within the remaining time.
  - Make the app show real song metadata and real art now, while still preserving a future path where fuller pre-organized data can be generated by the tools and used later if that architecture is still wanted.

Why this switch happened:
- Still plan to finish doing full runs with my `v2` data getter in the future.
- Still want future iterations of the app to have the option of using more fully pulled/organized song data rather than depending only on live enrichment.
- However, for this phase of the project the direction deliberately changed because:
  - time left in the project was becoming a real constraint,
  - Am already comfortable working with external APIs directly,
  - Deezer already exposes most of the extra display metadata immediately needed,
  - the app did not actually need every possible extra field pre-written into the database in order to improve the user-facing experience right now,
  - and preview URLs were always going to be awkward to treat as permanent DB fields anyway because they expire.
- So the practical decision became:
  - for now, use the live external Deezer API instead of requiring a full “pull everything with my `v2` tool and write it into the database first” workflow,
  - but keep the `v2` tool untouched because it is still expected to matter for future iterations.

What this pivot means and what it does not mean:
- It means:
  - the app now resolves most additional song display metadata live when songs are actually needed on screen.
- It does not mean:
  - Abandoned my `v2` tool,
  - No longer care about fuller organized data,
  - or the conclusion was that live-only enrichment is the final forever architecture for the project.
- This was a milestone-driven implementation pivot, not a “delete the old direction and pretend it never mattered” change.

What explicitly did not change:
- Did not rewrite or delete the tools.
- Kept `tools/` untouched on purpose.
- Did not convert the app to use lyrics pulling in this route.
- Did not implement language extraction in this route.
- Did not bulk-insert all of the new Deezer metadata into the shared project database.
- Did not treat preview URLs as permanent truth because they expire.
- Did not move Deezer API calling into the frontend.

Core architecture after the pivot:
- The project database is still the source of the base song lists.
- The backend still calls the existing stored procedures to get those base song rows.
- After the backend gets those rows, it enriches each song on demand with Deezer data.
- The frontend still talks to the app backend only.
- The frontend does not talk to Deezer directly.
- The backend caches successfully resolved Deezer metadata locally so repeated requests can be answered much faster after the first resolution.
- The frontend receives song objects that now contain:
  - the original DB-driven song values,
  - plus the extra Deezer metadata the backend was able to resolve for that song.

Why the backend owns Deezer instead of the frontend:
- Keeps external API logic out of browser/mobile code.
- Keeps app-side secret handling centralized.
- Gives me one place to rate-limit, retry, cache, and fall back safely.
- Lets web and mobile both benefit from the same enrichment logic through normal app endpoints.
- Prevents the frontend from having to orchestrate multiple raw Deezer requests for every song card/list item.

How the backend resolves a song now:
1. A screen asks the backend for a country/profile/hidden-gem song payload.
2. The backend gets base song rows from the existing stored procedure.
3. For each song row, the backend tries to resolve Deezer metadata using song name + artist name.
4. Before hitting Deezer, it checks the local cache first.
5. If the song is already cached and still usable, the backend reuses that cached metadata.
6. If it is not cached or needs refresh, the backend searches Deezer and then enriches the song with the resolved data.
7. The resolved metadata is written back into the local cache so later requests can reuse it.
8. The enriched song object is returned to the frontend.

Important unresolved-song rule:
- If the backend cannot resolve a usable Deezer result for a song, that song is not returned as a “broken” item.
- Instead:
  - it is skipped,
  - and the backend continues farther into the same stored-procedure result stream to try to fill the visible UI count with usable songs.
- This was done intentionally so the user would not see broken song rows with empty Deezer-dependent metadata.

Unresolved retry protection:
- Unresolved songs are not hammered repeatedly.
- The service has:
  - `UnresolvedRetryWindow = 12 hours`
- Meaning:
  - if a song was just marked unresolved, the backend will not immediately keep re-trying it on every request within that window.

Cache design:
- Local cache folder:
  - `backend/Capstone.API/live_song_enrichment_cache/`
- Files written there:
  - `live_song_cache.json`
  - `live_song_cache.csv`
  - `live_song_cache.txt`
- That folder is intentionally gitignored:
  - `.gitignore` includes `backend/Capstone.API/live_song_enrichment_cache/`

What each cache file is for:
- `live_song_cache.json`
  - the main working cache snapshot
  - this is the file the backend loads into memory
- `live_song_cache.csv`
  - readable export of resolved entries
  - useful for inspection / scanning / spreadsheet-like review
- `live_song_cache.txt`
  - readable block-style export of resolved entries
  - useful when quick plain-text inspection of the data is needed

How the cache works in practice:
- On first service use, the backend loads `live_song_cache.json` into memory as `_cacheByKey`.
- After that, normal cache hits are in-memory lookups first.
- That means the backend is not re-reading the file from disk for every single song request.
- After new data is resolved, the service persists the updated snapshot back out to JSON/CSV/TXT.

Why this matters for speed:
- A cache hit is much faster than a live Deezer call.
- The true slow path is:
  - first-time enrichment,
  - preview refresh,
  - or a Deezer retry/fallback path during transient failures.
- So the design goal became:
  - cached songs should feel fast,
  - uncached songs can temporarily show loading UI,
  - and the app should never feel empty just because live enrichment is still happening.

Preview URL handling:
- Preview URLs are deliberately treated differently from the rest of the metadata.
- Reason:
  - the preview URL itself expires
  - so it cannot be treated as permanent truth the same way the rest of the metadata can
- Current preview behavior:
  - store the preview URL when resolved
  - parse its expiry time
  - reuse it while it is still valid
  - refresh it by known Deezer track id when it expires
  - avoid serving an obviously expired preview as if it were still fresh

Exact preview expiry parsing rule:
- The backend reads the `hdnea` query parameter from the preview URL.
- It looks inside that value for the `exp=` segment.
- That Unix timestamp is converted into `PreviewExpiresAtUtc`.

What “keep preview URLs temporarily” means in this implementation:
- If the user plays the same song again and the preview URL is still valid:
  - reuse it from cache
  - do not re-ping Deezer unnecessarily
- If the preview is expired:
  - refresh it through the backend
- If refresh fails and the old preview is expired:
  - do not keep pretending the expired preview is valid

Implemented rate-limit rule:
- Treat Deezer as:
  - `50 requests every 4.9 seconds`
- Code constants:
  - `DeezerMaxRequests = 50`
  - `DeezerWindow = 4.9 seconds`
- Enforcement mechanism:
  - sliding-window rate limiter in the enrichment service

Reason `4.9` was used instead of `5.0`:
- The goal was to stay slightly inside the limit rather than riding the exact edge of it.

Failure/retry hardening added after real backend errors:
- Real issue encountered during testing:
  - Deezer transport/TLS requests were sometimes failing with:
    - `HttpRequestException`
    - `IOException`
    - `SocketException (54)`
    - `Connection reset by peer`
- Problem before hardening:
  - those transient external failures could bubble up and break country-song endpoint responses
- Hardening added:
  - retry transient Deezer failures up to `3` attempts
  - retry delays:
    - `250ms`
    - `800ms`
    - `1600ms`
  - treat `404` as a normal “not found / no data” outcome
  - if refresh/search fails transiently and there is already cached resolved data:
    - return safe cached metadata instead of failing the whole request
  - if fallback cache still contains an expired preview:
    - strip that expired preview from the safe fallback entry instead of serving a dead preview URL

UI/loading philosophy locked in:

  - use the existing placeholder colors where art is not ready yet
  - put a centered spinner on the art slot while the art is loading
- In other words:
  - visible placeholders and progressive loading were preferred over empty holes and stalled-looking sections

Problem with the first live-enrichment version:

Visible-only loading / scroll-driven strategy:
- Explicitly did not want the app to request metadata for songs the user could not even see yet.
- So the desired rule became:
  - initial render should use the first visible content that is already available

- This was meant to match the same philosophy already used to avoid overloading the database with giant initial song pulls.

```sql
USE HiddenGemMusic;
GO

CREATE OR ALTER PROCEDURE sp_GetDiscoverPageInfo
    @Year INT
AS
BEGIN
    SET NOCOUNT ON;

    SELECT
        c.iso_code AS country_code,
        c.full_name AS country_name,
        c.latitude,
        c.longitude,
        c.region,
        ISNULL(hg_agg.hidden_gem_count, 0) AS hidden_gem_count,
        top_song.top_album_name AS top_album_name,
        top_song.top_artist_name AS top_artist_name
    FROM Country c
    LEFT JOIN (
        SELECT
            country_id,
            COUNT(*) AS hidden_gem_count
        FROM HiddenGems
        WHERE chart_year = @Year
        GROUP BY country_id
    ) hg_agg ON hg_agg.country_id = c.country_id
    LEFT JOIN (
        SELECT
            ce.country_id,
            COALESCE(NULLIF(s_inner.album_name, ''), s_inner.title, 'Unknown Song') AS top_album_name,
            COALESCE(NULLIF(a_inner.artist_name, ''), 'Unknown Artist') AS top_artist_name,
            ROW_NUMBER() OVER (
                PARTITION BY ce.country_id
                ORDER BY COUNT(*) DESC, ce.song_id ASC
            ) AS rn
        FROM ChartEntry ce
        JOIN DIM_Song s_inner ON s_inner.song_id = ce.song_id
        LEFT JOIN Bridge_SongArtist bsa_inner
            ON bsa_inner.song_id = s_inner.song_id
           AND bsa_inner.artist_order = 1
        LEFT JOIN DIM_Artist a_inner ON a_inner.artist_id = bsa_inner.artist_id
        WHERE ce.country_id IS NOT NULL
          AND YEAR(ce.snapshot_date) = @Year
        GROUP BY ce.country_id, ce.song_id, s_inner.album_name, s_inner.title, a_inner.artist_name
    ) top_song ON top_song.country_id = c.country_id AND top_song.rn = 1
    WHERE c.latitude IS NOT NULL
      AND c.longitude IS NOT NULL
      AND c.iso_code IS NOT NULL
      AND LEN(c.iso_code) = 2
      AND c.iso_code NOT LIKE '%[^A-Z]%'
    ORDER BY c.full_name;
END;
GO
```
